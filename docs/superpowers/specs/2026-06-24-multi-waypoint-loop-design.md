# 多点路点闭环循环设计

## 目标

在现有多点路点导航基础上增加闭环循环模式。

路点为 `P1、P2、P3、P4` 时，单圈执行顺序为：

`P1 -> P2 -> P3 -> P4 -> P1`

支持两种停止方式：

- 指定圈数：完成设定圈数后自动停车。
- 持续循环：一直运行，直到手动停止、报警或其他安全停车条件触发。

现有单次导航模式继续保留。

## 执行规则

### 单次执行

按现有逻辑依次执行 `P1 -> P2 -> P3 -> P4`，到达最后一个路点后停车。

### 闭环循环

1. 首次启动时，从机器人当前位置前往 `P1`。
2. 到达 `P1` 后，依次执行 `P1 -> P2 -> P3 -> P4 -> P1`。
3. 返回 `P1` 时记为完成一圈。
4. 如果未达到指定圈数，直接开始下一圈的 `P1 -> P2`。
5. 达到指定圈数后停车，并将任务状态设置为完成。
6. 持续循环模式不限制圈数，只响应手动停止和安全停车条件。

每段运动继续沿用现有流程：

`读取当前 RTK -> 计算目标航向 -> 下发转向 -> 确认航向到位 -> 直行到目标点`

## 启动参数

`POST /vehicle/goToPoints/start`

请求参数：

```json
{
  "loop": true,
  "loopMode": "count",
  "loopCount": 3
}
```

字段说明：

- `loop`：是否启用闭环。
- `loopMode`：`count` 表示指定圈数，`continuous` 表示持续循环。
- `loopCount`：指定圈数模式下的目标圈数，必须大于 0。

未传参数时按照现有单次执行模式处理，保证旧前端兼容。

## 运行状态

Redis 增加以下状态：

- `waypointLoopEnabled`：是否启用闭环。
- `waypointLoopMode`：`count` 或 `continuous`。
- `waypointLoopTarget`：目标圈数，持续循环时为 `0`。
- `waypointLoopCurrent`：已完成圈数。
- `waypointIndex`：当前目标路点索引。
- `waypointTotal`：路点总数。

`GET /vehicle/goToPoints/progress` 返回：

```json
{
  "running": true,
  "currentIndex": 2,
  "total": 4,
  "loop": true,
  "loopMode": "count",
  "currentLoop": 1,
  "targetLoop": 3
}
```

## 停止与异常

- 手动调用停止接口后立即停车并退出循环。
- 转向失败、目标点执行失败时停止整个任务，不继续下一点或下一圈。
- 车辆报警、停车信号和现有安全保护继续拥有最高优先级。
- 路点少于两个时不允许启动闭环模式。
- 运行过程中不读取重新编辑后的路点列表，使用启动时的路点快照。
- 停止或完成后清理闭环运行状态，但保留路点列表。

## 前端结构

多点导航面板增加：

- 执行方式：单次执行、闭环循环。
- 循环方式：指定圈数、持续循环。
- 圈数步进输入框，仅在指定圈数时显示。
- 闭环提示：`P1 -> P2 -> ... -> P1`。
- 运行状态：当前圈数、目标圈数、当前目标路点。

执行中锁定路点增删、排序、执行方式和圈数设置，保留停止按钮。

手动输入经纬度区域从主界面移除，仅保留“记录当前 RTK 为路点”。

## 测试范围

- 单次模式保持原有执行顺序。
- 指定圈数模式能够准确完成设定圈数。
- 持续循环能够反复回到 `P1` 并继续下一圈。
- 返回 `P1` 后只增加一次圈数。
- 手动停止能够在任意路段退出。
- 转向或直行失败后不会进入下一路点。
- 少于两个路点时拒绝闭环启动。
- 进度接口正确返回当前路点和圈数。
- 前端正确提交模式参数并展示运行进度。
